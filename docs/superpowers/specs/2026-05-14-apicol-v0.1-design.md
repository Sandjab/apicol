# Design — apicol v0.1.0 (implémentation + publish PyPI)

**Date** : 2026-05-14
**Statut** : approuvé pour passer en planification d'implémentation
**Auteur** : brainstorming JP × Claude Code

## Objectif

Implémenter la totalité de la SPEC v0.1 documentée dans `SPEC.md` et publier le package `apicol` sur PyPI, avec un maximum de garde-fous (TDD, property-based testing, intégration gated, OIDC + TestPyPI).

Ce document est un **spec d'exécution** : il fige les décisions du brainstorming et sert d'entrée au plan d'implémentation produit par `writing-plans`.

## Décisions de cadrage

| Dimension | Choix retenu |
|---|---|
| Périmètre | SPEC complète v0.1.0 livrée en un seul shot (Client+AsyncClient, chat/achat sync+async, backends anthropic+litellm, anthropic_native, claude_cli_chat/achat, errors typées) |
| Nom PyPI | `apicol` — vérifié libre sur PyPI et TestPyPI le 2026-05-14 |
| Tests | pytest unit avec mocks + Hypothesis (PBT) sur traducteurs, coverage ≥95% lines+branches |
| Intégration | Smoke tests contre vraies APIs déclenchés uniquement sur tag `v*`, bloque la publish PyPI si fail |
| Publish | Trusted Publisher OIDC + gate TestPyPI avant prod + manual approval sur le job publish-pypi |
| CI matrix | Python 3.10, 3.11, 3.12, 3.13 sur ubuntu-latest |
| Style d'implémentation | TDD strict bottom-up séquentiel, un commit par module |
| Frontière TOS | `claude_cli_chat/achat` importés directement dans `__init__.py` depuis `_backends.claude_cli`, jamais accessibles depuis `_route` ou `Client` |
| Versioning | SemVer, source de vérité `pyproject.toml`, `apicol.__version__` lu via `importlib.metadata` |
| Licence | MIT (déjà dans `pyproject.toml`) |

## 1. Architecture et modules

```
src/apicol/
├── __init__.py          # Surface publique exclusivement (re-exports)
├── _errors.py           # ApicolError + sous-classes
├── _config.py           # Config (frozen dataclass) + load_from_env()
├── _client.py           # Client, AsyncClient + cache implicite
├── _route.py            # pick_backend(config) -> (sync_callable, async_callable)
└── _backends/
    ├── __init__.py
    ├── anthropic.py     # complete/acomplete + traducteurs OpenAI↔Anthropic
    ├── litellm.py       # complete/acomplete (pass-through)
    └── claude_cli.py    # complete/acomplete (subprocess wrapper)
```

### Responsabilités (boundaries)

| Module | Responsabilité unique | Dépendances internes |
|---|---|---|
| `_errors` | Hiérarchie d'exceptions custom | aucune |
| `_config` | Valider une config en un `Config` immutable, parser env vars | `_errors` |
| `_route` | Mapper `config.backend` → couple de callables | `_config`, `_backends.*` |
| `_backends/anthropic` | Traduire OpenAI↔Anthropic, appeler SDK `anthropic` | `_errors`, `_config`, `anthropic` |
| `_backends/litellm` | Pass-through vers `litellm.completion/acompletion`, injecter env vars | `_errors`, `_config`, `litellm` |
| `_backends/claude_cli` | Aplatir messages → subprocess `claude -p` → parser stdout | `_errors`, stdlib |
| `_client` | Encapsuler config + dispatcher + cache implicite | `_config`, `_route`, `_errors` |
| `__init__` | Re-exports publics + wrappers globaux minces | `_client`, `_backends.claude_cli` |

### Frontière TOS

`claude_cli_chat` et `claude_cli_achat` sont des fonctions globales **importées directement** dans `__init__.py` depuis `_backends.claude_cli`. Elles ne sont **jamais** atteignables depuis :
- `apicol.chat()` ou `apicol.achat()` (les wrappers globaux)
- `Client.chat()` ou `AsyncClient.chat()`
- `_route.pick_backend()` (qui rejette `backend="claude_cli"` même si reçu)

`_config.py` rejette `APICOL_TYPE=claude_cli` à `load_from_env()` avec un message explicite renvoyant vers `apicol.claude_cli_chat()`.

## 2. Data flow

### Chemin `chat()` global (mono-backend via env vars)

```
apicol.chat(messages, **kwargs)
  → _client._get_implicit_sync_client()
      → hash(APICOL_TYPE, APICOL_KEY, APICOL_MODEL, APICOL_URL)
      → inchangé : reuse cached Client
      → changé   : _config.load_from_env() → new Client → replace cache
  → client.chat(messages, **kwargs)
      → sync_callable(messages, config, **kwargs)
          (= _backends.{anthropic|litellm}.complete, résolu au __init__ du Client)
      → dict format OpenAI : {id, model, choices, usage}
```

### Chemin `Client.chat()` explicite (multi-backend)

Idem mais sans cache implicite (l'utilisateur a construit son `Client`).

### Chemin async

Symétrique strict : `apicol.achat()` → `AsyncClient` → `async_callable` (second élément du couple `pick_backend`) → `await ...`. Cache implicite sync et async sont **séparés** (slots distincts), bien que keyés sur les mêmes env vars.

### Frontière de traduction (backend Anthropic)

**Seul endroit** où l'on traduit. Toutes les autres couches parlent OpenAI.

**`_openai_to_anthropic(messages, **kwargs) -> dict`** :
1. Extraire le(s) message(s) `role="system"` → concaténer dans le kwarg `system`
2. Préserver `user`/`assistant`, normaliser `content` (str OU liste de blocks `{type:"text", text:"..."}`)
3. Mapper `reasoning_effort` ∈ {none, low, medium, high} → `thinking={type, budget_tokens}` selon table SPEC + détection modèle Opus 4.7 (mode `adaptive` au lieu de `budget_tokens`)
4. Default `max_tokens=4096` si absent (Anthropic requiert le champ) + log warning sur `logging.getLogger("apicol")`
5. `NotSupportedError` levée si `messages` contient `role="tool"` ou si kwargs contient `tools=` ou `stream=True`

**`_anthropic_to_openai(response) -> dict`** :
1. `response.content[0].text` → `choices[0].message.content`
2. `usage.input_tokens`/`output_tokens` → `prompt_tokens`/`completion_tokens` + somme
3. `stop_reason` mappé (`end_turn → stop`, `max_tokens → length`, …)
4. `id` et `model` préservés à l'identique

### Chemin LiteLLM (pass-through)

Aucune traduction sur les messages. Trois manipulations contrôlées :
1. Injecter `APICOL_KEY` dans le bon nom d'env var LiteLLM par provider détecté depuis `model` (ex. `openai/gpt-5` → `OPENAI_API_KEY`) — **seule magie env var acceptée**
2. Injecter `api_base=APICOL_URL` (ou `base_url` du Client) si présent
3. Wrapper toute `litellm.exceptions.*` en `BackendError` avec `__cause__` préservé

### Chemin `claude_cli` (totalement séparé)

```
claude_cli_chat(messages, *, model=None, timeout=120)
  → vérifier `which claude` (sinon BackendUnavailableError)
  → _flatten_messages_to_transcript(messages)
      → format transcript-style :
        "Human: <user_msg>\n\nAssistant: <assistant_msg>\n\nHuman: ..."
        (system messages préfixés en tête comme contexte)
  → subprocess: `claude -p <transcript> [--model X] --output-format json`
      avec timeout, capture stdout/stderr
      (flags exacts à confirmer via `claude --help` au moment de l'implémentation — cf. § Points à vérifier)
  → parse JSON → extraire content
  → dict format OpenAI minimal (usage=None, id="claude-cli-<uuid>")
```

## 3. Error handling

### Hiérarchie

```python
class ApicolError(Exception): ...          # racine, exposée publiquement
class ConfigError(ApicolError): ...
class BackendUnavailableError(ApicolError): ...
class BackendError(ApicolError): ...
class NotSupportedError(ApicolError): ...
```

### Règles

1. **Toute exception SDK est wrappée** en `BackendError` (ou `BackendUnavailableError` si setup). Aucune `anthropic.APIError` ou `litellm.exceptions.*` ne sort brute.
2. **`__cause__` toujours préservé** via `raise X(...) from original`.
3. **Pas d'auto-retry, pas de fallback automatique.** L'application orchestre.
4. **Erreurs levées le plus tôt possible** : `ConfigError` à la construction du `Client`, pas au premier `chat()`. `NotSupportedError` vérifiée avant l'appel SDK.

### Tableau récap

| Lieu | Erreur typique | Trigger |
|---|---|---|
| `_config.load_from_env()` | `ConfigError` | `APICOL_TYPE` absent/invalide/`claude_cli`, clé manquante hors cas local |
| `Client.__init__` | `ConfigError` | Combinaison invalide |
| `Client.anthropic_native()` | `BackendUnavailableError` | Appelée sur Client litellm |
| `_backends/anthropic.complete` | `NotSupportedError` | `role="tool"`, `tools=`, `stream=True` |
| `_backends/anthropic.complete` | `BackendError` | `anthropic.APIError` upstream |
| `_backends/litellm.complete` | `BackendError` | `litellm.exceptions.*` upstream |
| `_backends/claude_cli.complete` | `BackendUnavailableError` | `which claude` échoue |
| `_backends/claude_cli.complete` | `BackendError` | subprocess non-zero, timeout, JSON illisible |

### Logging

`logging.getLogger("apicol")` utilisé pour :
- WARNING : injection `max_tokens=4096` par défaut
- DEBUG : mapping `reasoning_effort` → `thinking`
- DEBUG : injection env var LiteLLM (nom de la var, **jamais** la valeur)

Aucun handler configuré par la lib.

### Invariants property-based (Hypothesis)

**Traducteur OpenAI→Anthropic** :
- Tout message OpenAI valide produit un payload Anthropic valide (jamais d'exception non typée)
- Messages `role="system"` toujours extraits dans kwarg `system`, jamais dans `messages`
- Ordre user/assistant préservé
- Contenu textuel invariant (pas de perte/duplication)
- `reasoning_effort=None` ⟹ pas de champ `thinking`
- Sur Sonnet, `reasoning_effort ∈ {low, medium, high}` ⟹ `thinking.budget_tokens` ∈ table SPEC
- Sur Opus 4.7, `reasoning_effort ∈ {low, medium, high}` ⟹ mode `adaptive`, pas de `budget_tokens`

**Aplatissement transcript-style (claude_cli)** :
- Prompt aplati contient tous les contenus textuels des messages
- Markers `Human:` / `Assistant:` dans l'ordre des rôles
- `messages=[]` ⟹ `ConfigError`

**Config parsing** :
- `APICOL_TYPE ∉ {anthropic, litellm}` ⟹ `ConfigError` (incluant `claude_cli`, vide, garbage)
- `(APICOL_TYPE=anthropic, APICOL_KEY=absent)` ⟹ `ConfigError`
- `(APICOL_TYPE=litellm, APICOL_MODEL=ollama/..., APICOL_URL=http://...)` sans clé ⟹ OK

**Cache implicite** :
- Deux `chat()` consécutifs sans modif env ⟹ même Client réutilisé
- Modifier `APICOL_MODEL` ⟹ nouvelle instance

### Mocks vs intégration

**Unit (par défaut, zéro réseau)** : `anthropic.Anthropic`, `anthropic.AsyncAnthropic`, `litellm.completion`, `litellm.acompletion`, `subprocess.run`, `asyncio.create_subprocess_exec` tous mockés.

**Integration** : `@pytest.mark.integration` skippé sans clés en env. Smoke minimal (un appel `chat()` qui retourne un texte, forme du dict vérifiée). Pas de tests de qualité de réponse (non-déterministe).

**claude_cli intégration** : `@pytest.mark.requires_claude_cli`, skippé sans `claude` dans PATH. En CI : toujours skip.

### Coverage gate

- ≥95% lines + branches sur `src/apicol/` (gate `fail_under=95` dans `pyproject.toml`)
- Exclusion ponctuelle via `# pragma: no cover` documentée
- HTML local non commité

## 5. CI/CD et release flow

### Trois workflows

```
.github/workflows/
├── ci.yml         # push/PR — gratuit, rapide, public
├── release.yml    # tag v* — publish PyPI gated
└── codeql.yml     # (optionnel) scan sécurité hebdo
```

### `ci.yml`

- Matrix : Python 3.10/3.11/3.12/3.13, ubuntu-latest
- Steps : checkout → uv → install → `ruff check` → `ruff format --check` → `mypy --strict` → `pytest --cov-fail-under=95`
- Codecov upload sur Python 3.12 uniquement
- Pas d'intégration, pas de secret consommé

### `release.yml` (déclenché sur tag `v*`)

Quatre jobs en chaîne :

1. **integration** (env `integration`, secrets `ANTHROPIC_API_KEY`/`OPENAI_API_KEY`)
   - Run `pytest -m integration` ; si fail → arrêt total
2. **build**
   - `uv build` → upload artifact `dist/`
3. **publish-testpypi** (env `testpypi`, OIDC)
   - Publish via `pypa/gh-action-pypi-publish@release/v1` vers `test.pypi.org`
   - Sleep 30s (propagation index) puis install depuis TestPyPI et `python -c "import apicol; print(apicol.__version__)"`
4. **publish-pypi** (env `pypi`, OIDC, **manual approval required**)
   - Publish vers PyPI prod

### Setup OIDC (une seule fois, hors code)

**Côté PyPI/TestPyPI** : créer le projet "pending" avec owner `Sandjab`, repo `apicol`, workflow `release.yml`, environments `pypi`/`testpypi`.

**Côté GitHub** : créer environments `integration`, `testpypi`, `pypi`. Sur `pypi` : activer **Required reviewers** (toi). Secrets dans `integration` : `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`.

### Flux end-to-end

```
1. Dev local : tests verts, ruff/mypy clean, bump version dans pyproject.toml + CHANGELOG
2. git tag v0.1.0 && git push --tags
3. release.yml démarre :
   ├─ integration → ok ?
   ├─ build
   ├─ publish-testpypi → install smoke ok ?
   ├─ ⏸ manual approval (env pypi)
   └─ publish-pypi
4. Création de la GitHub Release manuellement (notes copiées du CHANGELOG)
```

### Versioning

- SemVer strict
- `pyproject.toml` source unique de la version
- `apicol.__version__` lu via `importlib.metadata.version("apicol")`

## Points à vérifier à l'implémentation

Zones d'incertitude technique signalées par le brainstorming pour ne pas être enfouies :

- **Flags exacts de `claude -p`** : `--output-format json` et l'éventuel `--no-tools` (ou équivalent actuel) doivent être confirmés via `claude --help` au moment de l'Étape 5. Si la sortie JSON n'est pas accessible directement, fallback : parser le stdout texte avec un format conservateur.
- **Format de sortie `--output-format json`** : la forme exacte du JSON (champs `result`, `message`, `content`, etc.) sera inspectée au moment de l'implémentation. Le mapping vers le dict OpenAI minimal sera ajusté en conséquence.
- **`importlib.metadata.version("apicol")` en mode editable** : à vérifier que `pip install -e .` avec hatchling expose bien la version. Si KO, fallback : lire la version depuis un `_version.py` généré par hatchling.
- **Détection provider LiteLLM** (`_detect_provider_env_var(model)`) : la table de mapping doit couvrir au minimum `openai/*`, `gemini/*`, `anthropic/*` (via litellm), `ollama/*`, `lm_studio/*`, `vllm/*`, `openrouter/*`. Pour les autres providers, on tente l'injection via convention `{PROVIDER}_API_KEY` (uppercase) et on log un warning.
- **Compatibilité `pytest-asyncio` mode `auto`** avec Hypothesis : à valider que les stratégies Hypothesis fonctionnent dans des tests `async def`. Sinon, séparer property tests sync (Hypothesis) et async (pytest-asyncio classique avec exemples paramétrés).
